다음 지시사항을 반드시 모두 만족하는 논문 리뷰 글을 작성하라.

[전체 목표]
- 이 글은 논문을 “요약”하는 글이 아니다.
- 원 논문의 섹션 전개, 정의, 가정, 수식, 알고리즘, 실험 흐름을 최대한 그대로 따라가되,
  한국어로 자연스럽게 재서술한 “번역에 가까운 상세 리뷰 글”을 작성한다.
- 학부생/대학원생이 자신의 기술 블로그에 올리는 논문 리뷰라는 톤을 유지한다.
- 이 글만 읽어도, 해당 논문을 읽어본 적 없는 독자가 원 논문의 내용을 거의 완벽히 이해할 수 있어야 한다.
- 최소 A4 용지 5장의 분량은 나올 수 있게 매우 자세하게, 여러 문단으로 구분하여 작성해야 한다.

[구조 및 형식]
- 아래에 제시된 Markdown 템플릿 구조를 반드시 그대로 따를 것.
- 가능하면 원 논문의 섹션 구조(Introduction, Related Work, Method, Experiments, Ablation, Conclusion 등)를
  그대로 반영하고, 필요 시 ###, #### 단위로 세분화한다.
- 원문에 없는 섹션을 임의로 새로 만들지 말고, 기존 섹션을 확장하는 방식으로 서술한다.

[번역/서술 원칙]
- 직접 인용(quotation)은 꼭 필요한 경우에만 사용하며, 한 번에 1–2문장 이내로 제한한다.
- 대부분의 내용은 원문을 그대로 옮기기보다, 원문의 논리 전개를 따라가며 한국어로 풀어서 설명한다.
- 논문에 등장하는 모든 핵심 수식, 정의, 알고리즘, 가정은 가능한 한 빠짐없이 포함한다.
- 그림/표 처리(범용 규칙):
  - 기본: 논문 그림/표를 글에 “직접 첨부”하지 않고, **원 논문의 번호를 명시**해 텍스트로 재구성한다.
    - 그림: `(Fig. N: …)` 형식 (예: `(Fig. 2: LeNet-5 아키텍처 도식)`).
    - 표: `(Table X: …)` 형식 (예: `(Table I: C3의 부분 연결 스킴)`).
  - 선택: 글의 가독성을 위해 논문 그림/표 이미지를 **직접 삽입**할 수도 있다(Markdown 이미지 또는 HTML `<img>` 모두 허용).
    - 이 경우에도, 이미지의 바로 위/아래에 반드시 `(Fig. N: …)` / `(Table X: …)` 캡션을 함께 둔다(“어떤 원문 그림/표를 넣을지”가 번호로 추적 가능해야 함).

[수식/의사코드/정의]
- 중요한 기호와 변수는 처음 등장할 때 명확히 정의한다.
- 핵심 알고리즘은 반드시 의사코드(pseudocode) 블록으로 제시한다.
- 각 알고리즘/수식에 대해:
  (1) 수식적 정의
  (2) 직관적 의미
  (3) 왜 이런 형태가 필요한지
  를 모두 설명한다.
- LaTeX 수식에는 인라인의 경우 $를, 수식 블럭의 경우 $$를 반드시 사용한다.
- 중요한 주장이나 정리는 간단한 증명 스케치 또는 직관적 설명을 포함한다.

[헤딩 규칙]
- level 2 헤딩:
  - 기본: 원 논문 전개를 따라가는 본문 섹션은 "## 1️⃣", "## 2️⃣" 형식 사용
  - 예외(템플릿 고정 섹션): "## 💡", "## 👨🏻‍💻", "## ✅" 는 level 2로 허용
- level 3 헤딩:
  - 기본: 첫 번째는 "### 🔹 제목", 다음은 "### 🔸 제목", 이후 🔹/🔸를 교대로 사용
  - 예외(구분선 이후 Lucid 구현 파트의 “과정/단계”): "### 0️⃣", "### 1️⃣", "### 2️⃣" … 처럼 번호 기반 진행 헤딩을 사용해도 됨(이 경우 🔹/🔸 교대 규칙은 적용하지 않음)
- level 4 헤딩은 이모지 없이 텍스트만 사용 (#### 제목)

[Lucid 구현 파트에 대한 매우 중요한 지시사항]
- 구분선 이후의 “Lucid 구현” 파트에서는 **반드시 `lucid/models/` 디렉토리에 존재하는 실제 구현 코드를 읽고 이를 설명하라.**
- 이 파트의 목적은:
  `lucid/models/` 안에 이미 구현되어 있는 코드를
  “논문 관점에서 해석하고 단계적으로 설명하는 것”이다.

- (추가) 구현 요소 누락 금지:
  - “Lucid 구현” 파트에서는, 해당 논문/모델과 대응된다고 언급한 `lucid/models/` 내부 구현에서 **등장하는 모든 구현 요소(클래스, 함수, 팩토리 함수, `__init__`, `forward`, 주요 제어 흐름/루프, 레이어 구성 코드)** 를 빠짐없이 설명해야 한다.
  - 코드 스니펫은 `...` 등으로 핵심 구현을 생략하지 말 것. 길다면 여러 개의 스니펫으로 나누되, **최종적으로는 독자가 그 파일의 구현을 전부 따라갈 수 있을 정도로** 구성한다.

- 다음 사항을 엄격히 금지한다:
  ❌ 새로운 모델/레이어/연산 코드를 생성하는 행위
  ❌ 가상의 파일, 존재하지 않는 클래스나 함수 이름을 만들어내는 행위
  ❌ `lucid/models/` 외부의 임의 코드로 대체 설명하는 행위
  ❌ Lucid 프레임워크 내부 동작(autodiff, backward, gradient 전파, 내부 Tensor 구현)을 상세히 설명하는 행위

- 반드시 다음을 수행한다:
  1) 논문에서 제안한 핵심 아이디어가
     `lucid/models/` 내부의 어떤 파일, 어떤 클래스, 어떤 함수에 대응되는지 명시한다.
  2) 각 구현 단계마다:
     - 해당 코드가 논문의 어떤 수식, 알고리즘, 개념을 구현한 것인지 연결해서 설명
     - 왜 이런 구조로 구현되었는지 (모델링 및 아키텍처 설계 관점)
     - 논문 구조와 코드 구조가 어떻게 대응되는지 설명
     - 코드는 클래스 단위 혹은 함수 단위로 코드 블럭에 snippet 형태로 제공

- 이 파트의 목표는:
  “논문 속 개념이 실제 코드 구조에서 어떤 형태로 나타나는지를
   구현을 ‘읽는 관점’에서 이해시키는 것”이다.

[실험 및 결과 해석]
- 논문에 실험 섹션이 있다면:
  - 데이터셋, 메트릭, 학습 설정을 명확히 설명
  - 표에 나온 결과는 수치 기반으로 해석
  - Ablation이 있다면, 각 요소가 왜 중요한지 역으로 설명

[분량 및 완성도]
- 각 섹션은:
  문제 제기 → 이론적 배경 → 수식/알고리즘 → 직관 → 실험 해석
  의 흐름을 유지하며 충분히 상세히 작성한다.
- 리뷰 전체가 하나의 긴 학습 자료처럼 읽히도록 연결성을 유지한다.

[출처]
- 글의 맨 마지막에 “#### 📄 출처” 섹션을 둔다.
- MLA Citation 형식으로 논문 출처만 단독으로 기재한다.
- 저자, 논문 제목, 학회/저널, 연도, arXiv 또는 DOI(가능하면), URL(가능하면)을 포함한다.

[워크플로우: 논문 수집/텍스트화(새 채팅에서도 재현 가능)]
- 목표: 원문을 “읽고(섹션/수식/그림/표/알고리즘/실험) → 텍스트로 추출 → 리뷰 템플릿에 반영”까지 일관된 절차로 진행한다.
- 저장 위치:
  - 원문 PDF 및 중간 산출물은 `review/_sources/` 아래에 저장한다.
  - 최종 리뷰는 `review/papers/` 아래에 저장한다.

1) 원문 PDF 확보
   - 가능하면 공식 URL에서 다운로드하고, 파일명은 논문을 식별 가능하게 저장한다.
   - 예시:
     - `curl -L -o review/_sources/<paper>.pdf <pdf_url>`

2) (우선) PDF 텍스트 추출 시도
   - PDF에 “텍스트 레이어”가 있으면 OCR 없이 바로 추출하는 것이 가장 정확하다.
   - 도구/라이브러리(택1):
     - `pdftotext` (설치 필요할 수 있음)
     - Python: `pdfminer.six`
   - 주의: PDF 폰트/인코딩에 따라 텍스트가 깨질 수 있다(예: `(cid:...)`).

3) 텍스트 추출이 깨질 경우: 페이지 렌더링 + OCR로 텍스트화
   - (LeNet 리뷰 작성 시 실제 사용한 구성)
     - Python: `PyMuPDF (fitz)` — PDF 페이지를 이미지로 렌더링
     - `tesseract` — OCR 엔진
     - Python: `pytesseract`, `Pillow` — OCR 호출/이미지 처리
   - 설치(예시):
     - `python -m pip install pdfminer.six pymupdf pytesseract pillow`
     - (conda) `conda install -y -c conda-forge tesseract`
   - 산출물:
     - 렌더 페이지 이미지: `review/_sources/{model-name}/<paper>-pages/p01.png` 등
     - OCR 텍스트: `review/_sources/{model-name}/<paper>-ocr.txt` (페이지 경계 `===== PAGE n / N =====` 포함 저장 권장)

4) OCR 텍스트를 “검토용 인덱스”로 사용해 원문을 읽기
   - OCR에는 오타가 섞일 수 있으므로, 다음을 병행한다:
     - 섹션 헤딩(I., II., A., B. 등) 탐지로 큰 구조 잡기
     - 수식 번호/그림 번호(Fig. N, (k) 등)를 기준으로 PDF/페이지 이미지로 역추적해 확인
   - 목표는 “요약”이 아니라 “원 논문 전개를 따라가는 상세 리뷰”이므로, 섹션 순서를 유지한다.

5) Lucid 구현 파트 연결
   - `lucid/models/`의 실제 구현 코드를 직접 읽고, 논문 개념을 코드 구성 요소에 대응시킨다.
   - 코드 스니펫은 생략(`...`) 없이, 구현 요소를 빠짐없이 다룬다(상단 Lucid 규칙 준수).
